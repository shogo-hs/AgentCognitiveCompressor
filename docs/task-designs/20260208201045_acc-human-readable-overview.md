## 0. TL;DR
- 現状 README は起動手順中心で、ACC が「何を解決する仕組みか」の説明が不足している。
- 人間向けの導入として `README.md` に「ACCとは」「どう動くか」を追加する。
- 既存のセットアップ/起動/テスト導線は維持し、読者が最初に概念を理解できる構成へ更新する。
- 変更はドキュメントのみで、コード・API挙動には影響しない。

## 1. 背景 / 課題
- 依頼者から「ACCが何で、どんな仕組みか分からない」というフィードバックがあった。
- 現在の README は「できること」と実行手順が主で、ACC の目的・コアループ・CCS の位置づけが初見で把握しづらい。
- 人間向け入口は README であり、ここに概念説明を追加するのが最短導線。

## 2. ゴール / 非ゴール
### 2.1 ゴール
- README 単体で、ACC の目的と仕組み（Recall → Qualification → Commit → Decide）を理解できる。
- ACC の中核概念（CCS, Artifact, 有界メモリ）を平易な日本語で説明する。
- 詳細仕様は既存ドキュメントへリンクして深掘り可能にする。

### 2.2 非ゴール
- ACC アルゴリズム実装や API 仕様の変更。
- 新規アーキテクチャ文書の大規模追加。
- 論文相当の厳密な数式展開の全文再掲。

## 3. スコープ / 影響範囲
- 変更対象: `README.md`
- 影響範囲: リポジトリ初見読者の理解導線。
- 互換性: 破壊的変更なし（文書更新のみ）。
- 依存関係: 既存ドキュメント `docs/blue-print/arxiv-2601.11653.md` への参照。

## 4. 要件
### 4.1 機能要件
- README に ACC の目的を1段落で明記する。
- README に ACC 1ターンの処理フローを箇条書きで記載する。
- README に CCS の役割（永続状態・有界・置換コミット）を短く説明する。
- README に「どこを見れば詳細が分かるか」のリンクを追加する。

### 4.2 非機能要件 / 制約
- 日本語で簡潔に記述する。
- README の既存起動手順・テスト手順は維持する。
- 専門用語は必要最小限にし、初見読者が読める粒度にする。

## 5. 仕様 / 設計
### 5.1 全体方針
- README 冒頭に「ACCの概要」セクションを追加し、読む順番を「概要 → できること → セットアップ」にする。
- 概要では「課題（長期対話での記憶劣化）」「解法（有界状態への圧縮コミット）」「実装上の流れ（4ステップ）」を示す。
- 詳細は既存 blue-print へリンクし、README 自体は導入に特化する。
